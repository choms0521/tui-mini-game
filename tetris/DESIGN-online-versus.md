# Tetris 온라인 대전 설계서

터미널 Tetris에 1:1 온라인 대전을 붙이기 위한 설계 문서다. 구현 착수 전 단계의
청사진이며, 실제 코드는 포함하지 않는다(새로 추가할 함수의 의사 시그니처만 제시).

상태: 설계 검토용 초안 (구현 미착수)
대상: `tetris/` (mini-game 워크스페이스)


## 1. 개요

### 목표
- 멀리 떨어진 두 사람이 1:1로 동시에 플레이하는 versus 대전.
- 줄을 지우면 상대에게 쓰레기 줄(garbage line)을 보내 방해하는 진짜 대전 규칙.
- 현재의 터미널(blessed) 클라이언트를 그대로 유지.
- 인터넷 너머 연결은 Tailscale을 전제로 한다(포트포워딩·클라우드 서버 불필요).

### 비목표 (이번 범위 밖)
- 3인 이상 다인전, 매치메이킹 서버, 순위표, 관전 기능.
- 웹 클라이언트 포팅.
- 롤백 넷코드 등 프레임 단위 지연 보정.

### 핵심 설계 결정 요약
| 결정 | 선택 | 근거 |
|---|---|---|
| 시뮬레이션 권위 | 각 peer가 **자기 보드만** 시뮬레이션, 상대 보드는 표시 전용 스냅샷 | desync 원천 차단, 입력 지연 0 |
| 동기화 단위 | 상대 보드 스냅샷 + 쓰레기 줄 이벤트 + 시드 | 보드 전체를 매 프레임 동기화할 필요 없음 |
| 블록 순서 | 양쪽 동일 시드(공정성 장치) | 동일한 7-bag 순서로 공정 대전. 단, 시드는 **동기화 수단이 아니다** |
| 연결 방식 | TCP P2P (한쪽 listen, 한쪽 connect) | 메시지가 작고 신뢰성이 필요하므로 TCP가 적합 |
| 동시성 | 수신 전용 스레드 + thread-safe 큐, 메인 루프가 매 tick drain | 기존 blocking 입력 루프를 최소 변경 |
| 메시지 인코딩 | JSON line 프로토콜(`\n` 구분) | 디버깅 쉽고 의존성 없음 |


## 2. 핵심 설계 원칙 — 두 축을 분리한다

대전 네트워킹에서 흔히 뒤섞이는 두 개념을 명확히 분리한다.

### 축 A — 연결 토폴로지 (누가 소켓을 여는가)
- **호스트(host)**: 포트를 `listen`하는 쪽.
- **게스트(guest)**: 호스트 주소로 `connect`하는 쪽.
- 이것은 순수하게 소켓 개설 순서의 문제일 뿐이다. Tailscale이 NAT을 투명하게
  통과시키므로 코드는 일반 LAN 연결과 동일하다.
- **호스트라고 해서 두 보드를 다 계산하지 않는다.** 연결을 받을 뿐이다.

### 축 B — 시뮬레이션 권위 (누가 상태를 계산하는가)
- 각 peer는 **자기 자신의 `GameState`만** 권위 있게 진행한다.
- 상대의 보드는 수신한 스냅샷으로 **그리기만** 한다(표시 전용, 로직 없음).
- 두 보드가 한 프로세스에서 함께 굴러가는 일은 없다.

### 왜 lockstep(시드만 공유하고 입력만 교환)이 아닌가
이론상 "양쪽이 같은 시드 + 같은 입력이면 화면이 일치한다"는 맞다. 그러나 현재
중력(gravity)은 `main.py`에서 `time.monotonic()` 실시간 시계로 진행된다
(`fall_interval(level)` 간격). 두 클라이언트의 벽시계는 정확히 같은 tick을 밟지
않으므로 lockstep은 미세한 어긋남(desync)에 취약하다.

스냅샷 모델은 이 문제를 통째로 우회한다. 내 보드는 100% 로컬에서 즉시 반응하고,
상대 보드는 약간 지연된 스냅샷이지만 "보는 용도"라 어긋나도 무방하다. **시드는
공정성 장치(같은 블록 순서)일 뿐, 동기화 메커니즘이 아니다.**


## 3. 네트워크 — Tailscale 위의 P2P

### Tailscale의 실제 역할 (과장 없이)
- Tailscale은 WireGuard 기반 메시 VPN이다. 같은 tailnet에 속한 두 기기는 NAT/공유기
  뒤에 있어도 `100.x.x.x` 대역의 가상 IP(또는 MagicDNS 이름)로 서로를 직접 본다.
- **Tailscale 전용 코드는 한 줄도 없다.** 호스트는 포트를 bind하고, 게스트는 상대의
  `100.x.x.x` 주소로 다이얼한다. 그저 NAT을 넘는 LAN 코드일 뿐이다.

### 정직하게 짚는 마찰점
- **tailnet 멤버십**: 두 사람이 같은 tailnet에 있어야 한다. 친구를 tailnet에
  초대하거나 node-sharing을 써야 하며, 게임을 켤 때마다는 아니고 최초 1회 설정이다.
- **DERP relay fallback**: 직접 경로(direct path) 협상이 실패하면 Tailscale의 DERP
  릴레이를 경유한다. 지연이 다소 늘지만 Tertis 대전에는 충분하다. zero-setup은 아니다.

### 연결 절차
```
[ 호스트 ]                                  [ 게스트 ]
 versus_main.py --host                       versus_main.py --connect 100.x.x.x:PORT
   socket.bind(("0.0.0.0", PORT))
   socket.listen()  ───────── TCP/WireGuard ───────►  socket.connect(...)
   accept()
        │
        └── HELLO 교환(시드 합의) → READY → 대전 시작
```


## 4. 동시성 모델 — 가장 큰 통합 리스크

네트워크 자체보다 **기존 입력 루프와 네트워크 I/O의 공존**이 진짜 위험 지점이다.

현재 `main.py`의 루프는 `term.inkey(timeout=POLL_TIMEOUT)`로 **블로킹 폴링**한다.
여기에 소켓 수신을 끼워 넣어야 한다.

### 채택안 — 수신 스레드 + thread-safe 큐
```
[ 수신 스레드 ]                         [ 메인 루프 (기존 구조 유지) ]
 socket.recv() 루프                       매 tick:
   → 줄 단위로 JSON 파싱                     1) term.inkey() 로 내 입력 처리(로컬)
   → inbound_queue.put(msg)                 2) inbound_queue 를 drain:
                                                - STATE  → 상대 스냅샷 갱신
                                                - GARBAGE→ 내 보드에 쓰레기 줄 추가
                                                - GAMEOVER/BYE → 종료 처리
                                             3) 중력 tick
                                             4) 송신: 내 STATE/공격 이벤트 sendall
```
- `inbound_queue`는 `queue.Queue`(thread-safe). 메인 루프는 블로킹되지 않고 비울 만큼만
  꺼낸다(`get_nowait` 루프).
- 송신은 메인 루프에서 직접 `sendall`로 처리한다(메시지가 작아 별도 송신 스레드 불필요).
- 대안: 전체를 `asyncio`로 재작성. 더 깔끔하지만 기존 루프를 통째로 바꿔야 하므로
  1차 구현에서는 채택하지 않는다.


## 5. 동기화 데이터

### 5.1 공정 시드
- 호스트가 시드를 정해 `HELLO`로 보낸다. 양쪽 모두 `random.Random(seed)`로 게임을
  시작하므로 동일한 7-bag 순서를 받는다.
- `game.new_game(rng)`는 이미 `rng`를 주입받으므로 변경이 거의 없다.

### 5.2 상대 보드 스냅샷
- 상대에게 보여줄 내 상태 스냅샷을 보낸다: `grid`, `active`(위치/모양), `score`,
  `lines`, `level`, `game_over`.
- 전송 시점: **블록이 고정될 때(on-lock) + 주기적**(예: 0.2초마다). 10×20 = 200셀
  남짓이라 대역폭은 무시할 수준이다.
- 수신 측은 이 스냅샷을 그대로 보관했다가 그릴 뿐, 어떤 게임 로직도 돌리지 않는다.


## 6. 쓰레기 줄(garbage) 메커니즘

### 6.1 공격량 테이블
가이드라인 표준을 단순화해 적용한다(콤보/T-spin은 이번 범위 밖).

| 지운 줄 | 보내는 쓰레기 줄 |
|---|---|
| 1 | 0 |
| 2 | 1 |
| 3 | 2 |
| 4 (Tetris) | 4 |

### 6.2 구멍(hole) 위치는 공격자가 결정한다
- 누가 쓰레기 줄의 구멍 칸을 정하느냐가 결정론의 관건이다.
- **공격자가 구멍 컬럼을 정해 메시지에 실어 보낸다.** 받는 쪽은 그 컬럼만 비운 회색
  줄을 바닥에 쌓는다. 공유 RNG 결합이 없어 견고하다.
- 여러 줄을 한 번에 보낼 때 구멍 컬럼을 같게 할지(세로 통로) 줄마다 다르게 할지는
  정책으로 둔다. 1차 구현은 "한 번의 공격은 같은 구멍 컬럼" 으로 단순화한다.

### 6.3 적용 시점
- 받은 쓰레기는 다음 블록이 고정되는 순간 보드 하단에 추가한다(즉시 추가 대신
  타이밍을 한 박자 미뤄 현재 조작 중인 블록과 충돌하지 않게 한다).
- 상쇄(counter: 받기 전에 줄을 지우면 공격을 깎음)는 1차 범위 밖, 추후 확장.


## 7. 메시지 프로토콜

### 프레이밍
- 한 메시지 = JSON 객체 한 줄 + `\n`.
- 수신 측은 버퍼를 모으다가 `\n`으로 잘라 한 줄씩 파싱한다.

### 메시지 타입
| type | 방향 | 페이로드 | 설명 |
|---|---|---|---|
| `hello` | host→guest | `seed`, `protocol`, `name` | 시드·버전 합의 |
| `ready` | 양방향 | — | 시작 준비 완료 |
| `state` | 양방향 | `grid`, `active`, `score`, `lines`, `level`, `game_over` | 상대 표시용 스냅샷 |
| `garbage` | 양방향 | `rows`(개수), `hole_col`(또는 컬럼 배열) | 쓰레기 줄 공격 |
| `gameover` | 양방향 | `final_score` | 자기 보드 종료 통보 |
| `bye` | 양방향 | `reason` | 연결 종료 |

### 예시
```json
{"type":"hello","seed":482913,"protocol":1,"name":"general"}
{"type":"garbage","rows":4,"hole_col":7}
{"type":"state","score":1200,"lines":11,"level":2,"game_over":false,"grid":"...","active":{"name":"T","row":3,"col":4,"matrix":"..."}}
```
- `grid`/`matrix`는 컴팩트 인코딩(예: 각 줄을 문자열로 직렬화)으로 줄여 보낸다.


## 8. 파일 구조 및 코드 변경 계획

기존 파일의 불변(immutable) 철학을 그대로 따른다. 모든 새 전이도 새 객체를 반환한다.

### 새 파일
| 파일 | 책임 |
|---|---|
| `net.py` | 소켓 개설/연결, 수신 스레드, JSON line 송수신, `inbound_queue` |
| `versus.py` | 대전 규칙: 공격량 계산, 쓰레기 적용 조율, 상대 스냅샷 보관(불변 값) |
| `versus_main.py` | 대전 진입점·루프(`main.py`의 versus 버전), `--host`/`--connect` 인자 |
| `versus_render.py` | 두 보드(내 보드 + 상대 보드) 나란히 렌더링. `render.py` 재사용 |

### 기존 파일에 더할 것 (의사 시그니처)
```python
# board.py  — 쓰레기 줄을 바닥에 추가하고 위로 밀어 올린다 (불변: 새 Grid 반환)
def add_garbage(grid: Grid, rows: int, hole_col: int) -> Grid: ...
#   - 맨 위 rows 줄을 잘라내고, 바닥에 (hole_col만 빈) 회색 줄 rows개를 붙인다.
#   - 회색 쓰레기 셀은 기존 piece-name과 구분되는 별도 마커(예: "#")로 표시.

# game.py  — 지운 줄 수로 보낼 공격량을 계산 (순수 함수)
def attack_lines(cleared: int) -> int: ...
#   - SCORE_TABLE 옆에 ATTACK_TABLE = {0:0, 1:0, 2:1, 3:2, 4:4} 추가.
#   - _lock() 이 cleared 를 이미 알고 있으므로, lock 결과에 "이번에 보낼 공격량"을
#     실어 보내도록 _lock 반환을 확장하거나, versus.py 가 cleared 를 관찰해 계산.
```

### 렌더링 변경
- `render.py`의 `board_lines()`는 그대로 재사용 가능하다(임의의 `GameState`를 그림).
- `versus_render.py`에서 내 보드를 좌측, 상대 스냅샷 보드를 우측에 배치하고 가운데에
  점수/공격 게이지를 표시한다. 단말 폭 요구치가 커지므로 `_warn_if_small` 기준 상향.


## 9. 단계적 로드맵

각 단계가 그 자체로 동작하므로 중간에 멈춰도 산출물이 남는다.

### 1단계 — 핫시트(hot-seat), 네트워크 없음
- 한 화면에 두 보드를 나란히, **같은 시드**로 띄운다.
- 한 명은 화살표, 한 명은 WASD 등으로 같은 키보드에서 플레이(임시).
- 목적: `versus_render.py` 레이아웃과 두 `GameState` 병렬 진행 골격 검증.
- 네트워크·공격 없음.

### 2단계 — Tailscale 네트워크 연결 + 상대 보드 동기화
- `net.py` + 수신 스레드 + 큐 도입.
- `--host` / `--connect` 로 두 프로세스 연결, `hello`/`ready`로 시드 합의.
- `state` 스냅샷 교환으로 상대 보드가 화면에 보이게 한다.
- 아직 공격 없음. "둘이 각자 치는 모습이 서로 보인다"까지가 목표.

### 3단계 — 쓰레기 줄 공격 + 승패
- `attack_lines`, `add_garbage`, `garbage` 메시지 도입.
- 줄 삭제 → 상대에게 공격, 수신 시 다음 lock에 쓰레기 줄 추가.
- 한쪽 `game_over` → `gameover` 통보 → 승패 표시.

### 4단계 (선택) — 다인전(N명)으로 일반화
- 연결 구조를 허브 중계(스타형)로, 공격에 대상(target)을 도입하고, 화면에 상대
  썸네일을 여러 개 띄운다. 상세 설계는 아래 10절을 참고한다.
- 이 단계는 1~3단계가 끝난 뒤의 별도 트랙이며, 1:1 없이는 진행하지 않는다.


## 10. 다인전(N명) 확장 설계

> 이 절은 1:1(1~9절)이 완성된 뒤의 별도 확장 트랙이다. 처음 읽는 사람도 이해할 수
> 있도록 개념부터 차근차근 설명한다.

### 10.0 한눈에 보는 핵심

다인전이라고 해서 모든 것이 다시 어려워지는 것은 아니다. **무엇이 그대로이고 무엇이
바뀌는지**부터 분명히 하자.

- **그대로인 것 — 네트워크(Tailscale).** Tailscale 입장에서는 사람이 2명이든 6명이든
  전부 "같은 tailnet 안의 노드"일 뿐이다. 인원이 늘어도 연결 방식 자체는 1:1과 같다.
- **그대로인 것 — 시뮬레이션 권위.** 1:1과 똑같이, **각자 자기 보드만** 계산한다.
  남의 보드는 받은 스냅샷으로 그리기만 한다. 이 원칙은 인원과 무관하게 유지된다.
- **바뀌는 것 4가지** — 다인전의 난이도는 오직 이 네 가지에 있다.
  1. **누가 누구와 연결되는가** (연결 토폴로지) → 10.1
  2. **줄을 누구에게 보내는가** (공격 대상 정하기) → 10.2
  3. **화면에 몇 명을 어떻게 보여주는가** (렌더링) → 10.3
  4. **언제 누가 이기는가** (승패 규칙) → 10.4

즉 다인전은 "불가능한 일"이 아니라 "1:1을 일반화하는 일"이다. 다만 위 네 가지가
새로 붙으므로 1:1보다 분명히 큰 작업이고, 터미널 화면 한계 탓에 인원 상한이 있다.


### 10.1 연결 토폴로지 — 누가 누구와 연결되는가

여러 사람을 잇는 방법은 크게 두 가지다.

#### 방법 A — 풀 메시(full mesh): 모두가 모두에게 직접 연결
```
   A ───── B
   │ \   / │
   │   X   │
   │ /   \ │
   D ───── C
```
- 모든 사람이 나머지 전원과 직접 연결을 맺는다.
- 연결 개수 = N×(N−1)/2. (4명=6개, 6명=15개로 빠르게 늘어난다.)
- 각자 자기 스냅샷을 N−1명 모두에게 따로 보내야 한다.
- 장점: 중계자가 없어 단일 장애점이 없다.
- 단점: 인원이 늘수록 연결·전송이 급증하고, "누가 기준인가"가 모호해 참가자 합류·이탈
  관리가 번거롭다.

#### 방법 B — 허브 중계(스타형): 한 명이 중계자
```
        A
        │
   D ── H ── B          H = 허브(중계자). 플레이어를 겸할 수 있다.
        │
        C
```
- 한 명(호스트)이 **허브(hub)** 가 되고, 나머지는 허브에만 연결한다.
- 각자 자기 상태를 허브에 **한 번** 보내면, 허브가 그것을 나머지 전원에게 **방송**한다.
- 연결 개수 = N−1. 인원이 늘어도 완만하게 증가한다.
- 장점: 구현이 단순하고 참가자 관리(합류·이탈·id 발급)가 한곳에 모인다. 확장이 쉽다.
- 단점: 허브가 꺼지면 게임이 끝난다(단일 장애점). 허브에 트래픽이 모인다.

> **중요한 오해 방지:** 허브는 "메시지를 중계"할 뿐, 남의 보드를 대신 계산하지 않는다.
> 게임 로직은 여전히 각자 자기 기기에서만 돈다. 허브는 우체국이지 심판이 아니다.

#### 비교 및 권장
| 항목 | 풀 메시 | 허브 중계(권장) |
|---|---|---|
| 연결 수 | N×(N−1)/2 | N−1 |
| 참가자 관리 | 분산·복잡 | 허브 한곳에 집중·단순 |
| 단일 장애점 | 없음 | 허브 |
| 확장성 | 인원 늘수록 부담 | 양호 |

**권장: 허브 중계.** 구현이 단순하고 1:1 구조에서 자연스럽게 확장된다. 허브는 별도
전용 프로세스로 둘 수도 있고, 플레이어 중 한 명이 겸할 수도 있다.


### 10.2 공격 대상 — 줄을 누구에게 보내는가

1:1에서는 상대가 한 명뿐이라 고민이 없다. 다인전에서는 쓰레기 줄을 **누구에게
보낼지** 정하는 규칙이 필요하다. 대표적인 방식은 다음과 같다.

| 방식 | 설명 | 성격 |
|---|---|---|
| 무작위(random) | 매 공격마다 살아있는 상대 중 한 명을 임의로 | 가장 단순, 운에 좌우 |
| 되갚기(payback) | 최근 나를 때린 사람을 우선 공격 | 직관적·복수극, 구현 쉬움 |
| 위협 우선(badges) | KO를 많이 낸 강한 상대를 노림 (테트리스99식) | 전략적, 규칙 복잡 |
| 균등 분산(spread) | 공격량을 살아있는 전원에게 조금씩 | 무난하나 결정타 약함 |
| 수동 선택(manual) | 플레이어가 키로 대상 지정 | 조작 부담↑, 몰입↑ |

**권장 시작점: 무작위 또는 되갚기.** 가장 단순하면서도 재미가 보장된다. 구현상으로는
공격 메시지에 `target`(대상 플레이어 id) 필드를 더하기만 하면 된다. 허브 중계 구조라면
허브가 대상에게만 전달한다.


### 10.3 렌더링 — 화면에 몇 명을 어떻게 보여주는가

터미널은 화면이 좁다. 한 보드는 가로로 약 22칸(`B.WIDTH×2 + 테두리`)을 차지한다.
풀사이즈 보드를 여러 개 늘어놓기는 어렵다. 그래서 다음과 같이 배치한다.

- **내 보드는 풀사이즈로 왼쪽에**, 상대들은 **작은 썸네일**로 오른쪽에 격자 배치한다.
- 썸네일은 셀을 1칸 폭으로 압축한 미니맵(색만 표시, 고스트·테두리 생략)으로 그린다.

```
┌────────────┐   상대 보드(썸네일)
│            │   ┌────┐ ┌────┐
│   내 보드   │   │ B  │ │ C  │
│  (풀사이즈) │   └────┘ └────┘
│            │   ┌────┐ ┌────┐
│            │   │ D  │ │ E  │   ← 죽은 사람은 회색 처리
└────────────┘   └────┘ └────┘
   점수/공격 게이지        살아있는 인원: 4
```

- **현실적 인원 상한: 4~6명.** 그 이상은 썸네일조차 비좁아 터미널에서 비현실적이다.
  (테트리스99처럼 99명을 한 화면에 띄우는 것은 그래픽 기반 클라이언트가 필요하다.)
- `versus_render.py`가 이 레이아웃을 담당하고, 1:1의 2-보드 렌더는 "상대 1명"인
  특수한 경우로 흡수된다.


### 10.4 승패 — 언제 누가 이기는가

- **배틀로열 방식**을 기본으로 한다. 보드가 막혀 `game_over`가 된 사람은 **탈락**하고
  관전 상태로 전환된다. 마지막까지 살아남은 한 명이 승자다.
- 탈락한 사람의 썸네일은 회색으로 표시하고, 화면에 "살아있는 인원" 수를 보여준다.
- 최종 순위는 탈락한 역순(늦게 죽을수록 높은 등수)으로 매긴다.


### 10.5 메시지 프로토콜 변경점 (1:1 대비)

다인전에서는 "누가 보냈고 누구에게 가는지"를 메시지가 담아야 한다.

- 모든 메시지에 `from`(보낸 사람 id)을 추가한다.
- 공격 메시지(`garbage`)에 `target`(대상 id)을 추가한다.
- 새 메시지 타입:
  - `lobby` — 허브가 알리는 현재 참가자 목록(id·이름).
  - `eliminated` — 누군가 탈락했음을 통보.
  - `winner` — 최종 승자 통보.
- 허브가 합류 순서대로 플레이어 `id`를 발급한다.


### 10.6 코드 변경점 (1:1 대비)

| 파일 | 1:1 | 다인전 |
|---|---|---|
| `net.py` | 소켓 1개 | 허브는 N−1개 `accept`, 게스트는 허브 1개에 `connect`. 라우팅(`from`/`target`) 처리 |
| `versus.py` | 상대 1명 스냅샷 | `dict[id, snapshot]`로 다수 상대 보관 + 공격 대상 선택 함수 |
| `versus_render.py` | 두 보드 나란히 | 내 보드 풀사이즈 + 상대 썸네일 격자 |
| 프로토콜 | 단순 메시지 | `from`/`target` + lobby/eliminated/winner |


### 10.7 권고 순서

1. **1:1을 먼저 완성한다(1~9절).** 다인전의 토대다.
2. 다만 처음부터 `net.py`와 메시지에 **`from`/`target` 필드를 염두에 두고** 설계하면,
   나중에 허브 구조로 확장할 때 재작업이 거의 없다.
3. 그 위에 허브 중계 → 공격 대상 → 썸네일 렌더 → 배틀로열 순으로 얹는다.

## 부록 A — Tailscale 최초 설정 (사용자용 메모)
1. 두 사람 모두 Tailscale 설치 후 로그인(`tailscale up`).
2. 같은 tailnet에 합류(친구 초대 또는 node-sharing 승인).
3. 각자 `tailscale ip -4`로 자신의 `100.x.x.x` 주소 확인.
4. 호스트: `versus_main.py --host` (포트 listen).
5. 게스트: `versus_main.py --connect <호스트의 100.x.x.x>:<PORT>`.
6. 직접 경로 협상이 실패해도 DERP 릴레이로 자동 연결된다(지연만 다소 증가).
